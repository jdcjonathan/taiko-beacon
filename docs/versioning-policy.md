# Versioning Policy for Taiko Beacon Deliverables

This repo is the canonical source for Taiko's AEO-facing artifacts. All components specified in `AGENT_INSTRUCTIONS.md` are expected to have **versioned deliverables** in this repository so they can be cited with stable GitHub URLs.

## 1. Always-versioned deliverables

The following files and directories are treated as **source deliverables** and must be tracked in git:

- `packages/entity-pages/src/what-is-taiko.mdx`
- `packages/entity-pages/src/best-l2-for-ai-agents.mdx`
- `packages/context-pack/taiko-context.md`
- `packages/context-pack/AI.md.template`
- `packages/schema-injector/faq-seed.json`
- `packages/schema-injector/src/inject.ts`
- `packages/mcp-manifest/taiko-mcp-manifest.json`
- `packages/entity-corrections/corrections.json`
- `packages/entity-corrections/check-status.py`
- `packages/aeo-monitor/monitor.py`
- `packages/aeo-monitor/prompts.json`
- All dashboard source under `packages/aeo-monitor/dashboard/`

These files are part of the spec itself (pages, manifests, seeds, context packs, trackers) and must never be relied on as purely generated output.

## 2. Generated but versioned deliverables

Some artifacts are generated from scripts but are still considered canonical AEO assets and should be versioned:

- `packages/llms-txt/dist/llms.txt`
- `packages/llms-txt/dist/llms-full.txt`

These are generated by the `llms-txt` package from `docs/entity-definition.md` but are committed so:

- They have stable GitHub URLs for decks and documentation.
- Diff history shows how crawler-facing text evolved over time.

When adding new components, prefer this pattern for any generated asset that is referenced directly by external systems (LLMs, docs, or dashboards).

## 3. Ephemeral outputs

The only outputs that are intentionally **not** versioned are short-lived monitoring results:

- Individual monitor run JSONs under `packages/aeo-monitor/results/*.json`

These are time-series snapshots created by the AEO monitor. They may be rotated or stored elsewhere; they are not treated as canonical spec.

If long-term history is desired, either:

- Commit a summarised `history.csv` with an explicit retention policy, or
- Move results storage to a dedicated data bucket or repository.

## 4. .gitignore alignment

To keep noise out of git while preserving the deliverables above:

- Global ignores cover `node_modules/`, `dist/`, build artifacts, and `packages/aeo-monitor/results/*.json` / `*.csv`.
- Narrow exceptions are added for versioned generated assets inside otherwise ignored paths, e.g.:
  - `!packages/llms-txt/dist/`

When new components are added:

1. Decide whether their outputs are source deliverables, generated-but-versioned, or ephemeral.
2. Update this policy file with the new paths.
3. Adjust `.gitignore` in the same commit so deliverables are tracked but bulk build output stays ignored.
